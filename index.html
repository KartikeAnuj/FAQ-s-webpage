3. HTTP 401 Unauthorized-"code": "900901", "message": "Invalid Credentials"?
This error occurs when the auth token has expired. Auth token has a validity of 6 hours.
4. HTTP 401 Unauthorized- Response {“code": "900901”, “message": "Invalid Credentials”, “description": "Invalid JWT token. Make sure you have provided the correct security credentials"}
The old token becomes invalid once a new token is generated. Use the newly generated token. By using the updated token, you should be able to authenticate your requests successfully.
5. HTTP 404 Not Found {“code": "404”, “type": "Status report”, "message": "Not Found", "description": "The requested resource is not available."}
The access token in the Authorization header is used to authenticate the API that is being accessed. Hence use Authorization: Bearer
6. Do I need to use Authorization or Authentication in the header?
The old token becomes invalid once a new token is generated. Use the newly generated token. By using the updated token, you should be able to authenticate your requests successfully.
7. Do I need to generate the client-id, client-secret, session-id, and session-key over and over again to generate the auth token?
The short answer is no. Once the encrypted client-id, client-secret, session-id, and session-key are generated using the same values you can generate the auth token over and over again. However, please note that the token has an expiration time of 6 hours.
8. How to generate HMAC?
The steps to generate HMAC will be sent over mail. The hmac will be a base64 encoded string and the length of hmac will always be of length 44 irrespective of the size of the payload.
9. HTTP 403 Forbidden { "code": "900908”, "message": "Resource Forbidden", "description": "User is not authorised to access the resource."}?
You will have to subscribe this API in the API gateway. The steps to subscribing services is mentioned in Section 10.2 in Sandbox Access - SOP document placed in our Google Drive.
10. What is the encryptedAesKey and ivParameterSpec in the authorization and authentication API specification?
The encryptedAesKey is the session-key and the ivParameterSpec is the session-id.
11. What to do when I face an “invalid client-id value” and error- “E032”?
Always ensure that the client-id value that you have received during onboarding is encoded to base64 and sent in the header of the subscribed APIs. Example: ClientID: aQIKPTiOL73_g7hWO49upZvFaRst Base64 value: YVFJS1BUaU9MNzNfZzdoV080OXVwWnZGYVJzdA== Also, refer to other FAQs regarding client-id issues. You may face this issue when the auth token that is generated is different from the one being sent in the header of the subscribed API.
12. What is the meaning of the consent-id header?
The consent-id is a mandatory header which is an alphanumeric format that lenders need to generate. The logic is left up to the lender to generate the consent-id. This field is mandatory because it is to record that consent has been taken from the borrower to process the borrower's data.

For the ingress i.e., incoming traffic to the IPTPF platform from banks, whitelist the public IP 103.110.187.82 For the egress i.e., outgoing traffic from the IPTPF platform to banks, whitelist the two public IPs 103.110.186.137 & 103.110.187.111  Banks are also required to share their IPs which are to be whitelisted by the platform during onboarding to access the platform, api’s.

15. What is the platform-public-key.pem file mentioned in the Authentication and Authorization API Specification?
The platform-public-key.pem is the public key which is shared to the lender during onboarding. This file is used to generate the headers which are used in the auth API. The same public key is also used during hmac generation.

You should always send the base64 encoded client-id as a header. The encrypted client-id will be sent as a header-only during auth token generation.

27. How can I verify the integrity of the payload received?
To verify the integrity of the payload received. The platform offers a mechanism where the data is encrypted using SHA256withECDSA. The public key, signature and hmac is also found in the payload. The lender can use the public key and can verify the value of hmac received with the data in the response to ensure that the data hasn’t been tampered with. The source code to this can be requested over mail and will be shared over to the lender.

29.After successful registration of the lender on the platform, how to generate a bearer token to access APIs?
On successful registration, the lender will receive a User-Id and password to access the platform, along with the Client ID, Client Secret, and Public Key of the Platform. Using the received Client ID, Client Secret, and Public Key of Platform, the lender can refer and Authorization API Specification v1.0 shared with them to generate encrypted client Id , client Secret , Session Key and Session ID. Once encrypted client Id , client Secret , Session Key and Session ID is generated, Lender can invoke Authentication API to generate auth token (bearer token) to access the API they have already subscribed to.

38. How does the platform ensure data security?
The platform uses the highest level of data encryption. All data in rest and transit is encrypted using the most secure cryptographic algorithms. AES-256 is used for data encryption in rest. All the APIs are exposed as web services secured by TLS1.2 protocol having a minimum of RSA 2048-bit protection. As a policy, no PII (Personally Identifiable Information) will be persisted on the platform. However, in the case of asynchronous API calls, it may be needed to temporarily store data for up to 4 hours.

However, for this particular case, a trust store deployment restart is necessary, and this restart will affect two pods, specifically both Application manager gateway pods. This information will be valuable for future reference in similar situations involving SSL certificates and WSO2 deployments. 100. Why do we get an "E045: Missing mandatory header parameter " error even when all headers are sent by the lenders?
This error usually occurs when lenders are using the wrong endpoint with the correct headers. For example, the headers might be set correctly for the /getconsentformilkinsights endpoint, but the request is sent to a different endpoint, such as /getfarmermilkinsights. Ensure that the headers match the specific endpoint being used.
